# Security Policy

We take security seriously. If you've found a vulnerability, this
document explains how to report it so we can fix it without exposing
users in the meantime.

## Reporting a vulnerability

**Do NOT open a public GitHub issue.** Public issues are indexed
within minutes and turn an undisclosed bug into a known one.

Instead, email **security@humansheild.example** (replace with your
real address) with:

- A description of the issue.
- Reproduction steps. A minimal proof-of-concept is ideal — a
  curl/SQL/HTTP snippet, screenshot, or short video.
- The version, build hash, or URL where you observed the issue.
- Your contact info if you'd like attribution in the fix.

For high-severity issues (active exploitation, exposed secrets, mass
data exposure), you can also reach the on-call engineer via the
platform's status page contact form — mark the message **SECURITY**
in the subject so it bypasses the normal queue.

PGP key for sensitive reports: [link to public key when available].

## What you can expect

| Step | Owner | SLA |
|---|---|---|
| Acknowledge receipt of your report                       | Platform | 72 hours |
| Triage + assign severity (see table below)               | Platform | 5 business days |
| Initial mitigation in production                          | Platform | per severity |
| Post-fix verification + confirm with reporter             | Platform | within 7 days of fix |
| Public disclosure (CVE, advisory, release notes)         | Platform | coordinated with reporter |

We aim to keep you in the loop at every step. If we go silent for
more than 7 days after acknowledgement, please nudge.

## Severity definitions

| Severity | Definition | Mitigation SLA |
|---|---|---|
| **Critical** | Remote code execution, mass user data exposure, auth bypass, RLS escape, mass denial of service. | 24h |
| **High**     | User-to-user data exposure, privilege escalation within tenant, persistent XSS, broken access on sensitive endpoint. | 7 days |
| **Medium**   | Reflected XSS, CSRF on non-sensitive action, info leak that doesn't directly expose users. | 30 days |
| **Low**      | Best-practice deviations without practical exploit (missing security header on non-sensitive page, etc.). | 90 days, batched in normal release |

Severity is assigned by the platform team using the CVSS v3.1
calculator. We may disagree with your assessment; we'll explain why.

## Scope

### In scope

- The production web app at the primary public domain.
- The public-facing API (`/functions/v1/*` on the production Supabase
  project).
- The OAuth / authentication flow.
- The realtime channels for breaking-news + audit broadcast.
- The scraper service (Fly.io) endpoints reachable from the public
  internet.

### Out of scope

- **Findings against staging or preview environments.** Their access
  controls are intentionally relaxed so the platform team can
  develop — finding a relaxed CSP on a preview deploy is not a
  vulnerability.
- **Volumetric DoS / DDoS** against the public app. Anti-DoS is the
  CDN's job.
- **Social engineering** of team members.
- **Physical security** of team locations.
- **Spam / abuse reports** unless they enable user data exposure.
  Use the in-product abuse form for these.
- **Third-party services** we depend on (Supabase, Vercel, Fly.io,
  OpenRouter, etc.). Report those directly to the vendor; cc us if
  it affects our users.
- **Missing security headers without an exploit path.** We already
  enforce a strict CSP, HSTS, X-Frame-Options DENY, COOP, CORP, etc.
  (see [vercel.json](artifacts/humanproof/vercel.json)). If you find
  a header gap with no concrete impact, that's a Low-severity nice-
  to-have, not a critical.
- **Stack/version disclosure** (e.g. "this server returns
  `Server: nginx/1.x`"). We don't consider this a finding.
- **Reports generated by automated scanners with no manual
  validation.** We've seen the same Nessus report fifty times.

## What we promise

- We will not pursue legal action against good-faith security
  research that follows this policy. "Good faith" means: you don't
  exfiltrate more data than necessary to demonstrate the issue, you
  don't disrupt other users, and you give us reasonable time to
  fix.
- We will credit you in the fix advisory unless you ask us not to.
- We do not currently run a paid bug bounty. We may send a token of
  appreciation for high-impact reports — that's a thank-you, not a
  contract.

## What to do if you've already accidentally exposed user data

If you ran a test that returned data you shouldn't have — e.g. a
mass enumeration, or a query that returned another user's row —
**stop immediately**, delete any copies you may have made, and tell
us. We don't punish good-faith reports of accidents; we DO ask that
the response be honest and prompt.
