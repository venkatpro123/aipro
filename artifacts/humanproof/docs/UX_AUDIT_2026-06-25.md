# HumanProof — Premium UI/UX Audit

**Date:** 2026-06-25 · **Scope:** Visual design, information architecture, "Career OS" feel, personalization, mobile/dark-mode quality, premium polish. Functionality/API/backend correctness explicitly out of scope per audit brief.

**Method:** Live-driven evidence only. The dev server was run locally; a Playwright script drove the real 8-step Layoff Audit wizard end-to-end three times (companies: Microsoft, Netflix, Amazon, Tesla — real swarm pipeline, real OSINT/AI agent output, no mocks) to reach the actual V3 dashboard. All 12 top-level routes were screenshotted in desktop (1440×900), mobile (iPhone 13), dark, and light mode. Four parallel codebase research passes covered iconography/illustrations, mission/personalization logic, premium-polish/visual-storytelling wiring, and dark-mode/mobile CSS, each with file:line citations. Screenshots referenced below live in [`docs/ux-audit-screenshots/`](./ux-audit-screenshots/).

---

## Section 1 — Visual Design Audit

**Score: 78/100**

The marketing shell (home, pricing) and the dashboard's "hero" moments (score ring, Situation Brief) are genuinely premium — dark `oklch()`-based palette, a real typographic display font, restrained cyan/amber/emerald accent system, and a 195-custom-property design-token system (`src/index.css:4-292`). The score ring with animated arc, parallax tilt, and count-up (`SummaryTab.tsx:117-211`) would not look out of place in a Linear or Stripe product.

Two things undercut "expensive":
- **Hardcoded color drift.** 2,227 raw `rgba(255,255,255,…)` literals across 202 files bypass the token system entirely (worst: `StrategyTab.tsx` 49, `BehavioralIntelligencePanel.tsx` 43, `JobMarketLiquidityCard.tsx` 31). Premium products have *zero* tolerance for this — it's why Stripe's dashboard never looks "off" anywhere.
- **Two parallel icon-sizing conventions** (lucide `size=/strokeWidth=` props in nav vs. Tailwind `w-/h-` classes with no stroke override in dashboard panels) — see Section 4.

[01-home-desktop-dark.png](./ux-audit-screenshots/01-home-desktop-dark.png) shows the hero at its best: gradient headline, floating "Career Risk Radar" mockup card, live stat strip. It feels trustworthy and modern. It does **not** yet feel "enterprise AI OS" everywhere — several secondary pages (Section 2) read as placeholder-empty rather than minimalist.

The platform *can* tell stories visually, and largely already does inside the dashboard: `RiskTrendChart` (wired into `SummaryTab.tsx:51,949` and `AnalysisSummaryTab.tsx:15,323`), `SkillRadarChart`, `SkillDependencyGraph`, `CareerPathMap` (all wired into `ProtectionTab.tsx:103,127,131`) replace what would otherwise be prose. - Vesting/equity figures (`OverviewTab.tsx:1837,1843`) are bare `$XXXk` cells with no timeline, despite being inherently a "cliff → now → optimal-exit" story.

## Section 4 — Iconography Audit

**Score: 54/100** — the single most concrete, fixable finding in this audit.

The app has a disciplined primary icon system: **lucide-react, 194 files, zero react-icons usage.** But **107 files** also use raw emoji as icons, and the two systems collide in exactly the surfaces that matter most for first impressions and gamification:

| Location | Issue |
|---|---|
| `src/components/LayoffCalculator/LayoffInputForm.tsx:730-829` | Every wizard choice-card icon is an emoji (🌱📅💼🏆🎓📈🎯🌟✅⚠️🔄💎💻🤝📋🔬) — see [06-wizard-step3-radio-cards.png](./ux-audit-screenshots/06-wizard-step3-radio-cards.png), where 🌱📅💼🏆 sit as the only visual language on the most premium first-impression screen in the product |
| `src/services/achievementService.ts:35-48` | All 14 achievement icons are emoji strings, unrelated to the lucide system used in `AchievementGallery.tsx` itself (which imports `Lock` from lucide at line 9 but renders a literal `'🔒'` instead, line 87) |
| `src/components/AuditTabs/common/PhaseProgressSystem.tsx:227` | A lone `🔓` emoji sits inside an otherwise all-lucide file — lucide has the exact `Unlock` icon needed |
| `src/components/LayoffCalculator/LayoffCalculator.tsx:132-138` | A second, duplicate toast system uses `✓/✗/⚠/ℹ` glyphs while the app also ships `sonner` (already used at `App.tsx:774`) |

**Sizing is also forked**: nav/page-level icons use `size={18-20} strokeWidth={1.8}` (`App.tsx:358,420-443`); dashboard-panel icons (`SocialProofStrip.tsx`, `ReturnVisitPanel.tsx`) use Tailwind `w-2.5 h-2.5`/`w-3 h-3` with the lucide-default 2px stroke, so they read visually heavier/chunkier next to nav icons at a similar size.

**Recommendation:** a single afternoon's work — swap every emoji for its lucide equivalent (mapping already enumerated in the research evidence) and standardize on one sizing convention. This is the highest ROI, lowest-risk fix in the entire audit.

## Section 5 — Illustration Audit

**Score: 50/100**

12 illustration components exist in `src/components/illustrations/` (line-art + gradient-glow SVGs, animated pulse rings, on-brand). **Only 5 of 12 are used anywhere in the app** (`HomePage.tsx:4-8,319-334` uses 4 for marketing cards; `ProtectionTab.tsx:9,139` uses 1 for an empty state). **The entire `AdditionalIllustrations.tsx` file — 6 components — is dead code**: `NetworkIllustration`, `FinancialRunwayIllustration`, `LearningIllustration`, `InterviewIllustration`, `NegotiationIllustration`, `MilestoneIllustration` are imported nowhere.

This is the cheapest illustration roadmap possible — wire in assets that already exist:

| Opportunity | Current state | Use this existing, unused asset |
|---|---|---|
| Financial-runway panels | Numeric stat card only | `FinancialRunwayIllustration` |
| Negotiation intelligence panel | Text-only cards | `NegotiationIllustration` (literally named for this) |
| `ProtectionTab.tsx:18-39` skills list | Plain text | `LearningIllustration` |
| `ReturnVisitPanel.tsx:90` empty/first-return state | Returns `null` — blank | `MilestoneIllustration` |
| Wizard "institutional knowledge" choice cards | Emoji (see Section 4) | `NetworkIllustration` as a hero graphic, emoji removed |
| Locked achievement state | `'🔒'` emoji | At minimum the imported-but-unused lucide `Lock`; ideally a small custom illustration |

## Section 6 — Career OS Feeling Audit

**Score: 73/100**

This is the pleasant surprise of the audit. Once a user completes an audit, the dashboard does **not** read as a static report — it reads as a briefing from an analyst. Evidence, in order of appearance on the actual Netflix/Data-Analyst run captured live:

- A **Situation Brief** banner with a stated "priority action," not a stat dump ([07](./ux-audit-screenshots/07-dashboard-situation-brief.png))
- An **"AI Read"** card with a model-generated read of the situation and an expandable **"How the AI reasoned"** trace with a confidence percentage — this is the single best "intelligence platform" signal in the product
- **Career Preparedness** scored independently of layoff risk, with a named "weakest pillar" and a plain-English reason
- An **"AI Mission Brief (NEW)"** badge framing the situation/biggest-risk/priority in briefing language, with a completion counter (0/6) and a confidence percentage ([10](./ux-audit-screenshots/10-dashboard-ai-mission-brief.png))
- A **Career Contingency Plan** offering three named strategic paths (Protect & Strengthen / Negotiate & Capture / Transition & Advance) each with feasibility ranges, runway, and a cited success-rate source ([11](./ux-audit-screenshots/11-dashboard-career-contingency-plan.png))

This is consulting-memo-grade framing, not a dashboard. The gap is that this quality is **concentrated entirely in the post-audit dashboard** — Home, Risk Calculator, Products, Team, and Predictions (Sections 1-2) don't carry any of this voice, so the "OS" feeling currently only switches on after the 8-step wizard + ~60-75s wait. A first-time visitor on the homepage would not predict the quality of what's behind the CTA.

## Section 7 — Mission System Audit

**Score: 47/100**

The team built a real anti-repetition engine — credit due — but it is **disconnected from one of its own two UIs**, which is the kind of bug that, if a user hits it, reads as "this thing doesn't actually learn."

**What's real:** `selectWithRotation` (`actionPersonalizationEngine.ts:4099-4140`) excludes already-completed action IDs and tops up from adjacent risk tiers when a tier is exhausted. Re-running an audit changes `riskLevel`, which changes which action pool gets read — a genuine score→content loop. 1-2 star action ratings permanently suppress that action for the user (`actionCompletionService.ts:206-217`) and, at ≥10 cohort ratings with >40% negative, for everyone (`cohortFeedbackService.ts:27-46`).

**What's broken:** The app renders the action plan via **two parallel, disconnected UIs**. `PhaseProgressSystem.tsx:410` correctly calls `markActionComplete()`, feeding the rotation engine. But `ActionPlanTab.tsx` (rendered under "Analysis" view mode via `AnalysisActionsTab.tsx:368`) has its **own checkbox handler** (`ActionPlanTab.tsx:1999,2057`) that writes to a different localStorage key (`"actionPlanCompleted"`) and **never calls `markActionComplete`**. A comment at `v3/ActionsTab.tsx:640` ("ActionPlanTab removed — exact duplicate") shows the team believed this was already cleaned up; it wasn't. **A user living in Analysis mode will see the same action items indefinitely, even after checking them off** — exactly the "recycled intelligence" failure mode this audit was asked to look for.

Also visible live in the captured Action Roadmap ([12](./ux-audit-screenshots/12-dashboard-action-roadmap-debug-leak.png)): a card reading **"System Override Applied — The engine detected exceptional patterns: Signal conflict (critical): 1 discrepancies detected"** is rendered inline between two real action items, styled identically to them (same badges, same border treatment). This reads as an internal debug/QA message that leaked into user-facing copy — it should either be removed or rewritten in plain user-facing language; right now it actively breaks the "premium intelligence" illusion the rest of the page builds.

`monthlyActionPlanEngine.ts` (the calendar feature) takes zero history/completion parameters (`monthlyActionPlanEngine.ts:80-106`) — re-running an audit a month later will regenerate the identical Week 1-4 calendar.

The achievement system (Section 13) is a pure badge layer: unlocking one fires an event nobody listens to — it never changes future content (`achievementService.ts:81-84`, confirmed zero downstream call sites).

**Fix priority:** rewire `ActionPlanTab.tsx`'s checkbox to call `markActionComplete`, or finish the removal the code comment already claims happened. This single fix does more for "does the system evolve" than any new feature would.

## Section 8 — Personalization Audit

**Score: 55/100**

Genuine personalization exists (Section 7's rotation engine, score-tier re-reading, cohort-feedback suppression). But two specific "is this new intelligence?" moments are cosmetic:

- `ReturnVisitPanel` is wired with a real new-breaking-news-since-last-visit signal (`continuousIntelligenceService.ts:24-26,113-114`), but its `completedActions` and `marketChanges` props are **never passed** at the call site (`LayoffAuditDashboardV3.tsx:735-742`) — they silently default to `0`/`[]`. Two of the panel's three "what's new" sections are permanently dead regardless of what actually changed.
- A critical-risk user exhausts their action reservoir after one completion cycle for any role outside the ~6-8 hand-authored high-traffic roles (`actionPersonalizationEngine.ts:3195-3207`, a gap the code itself documents) and falls back to a generic 3-action "professional services" trio (`:4081`) — the exact "recycled intelligence" the brief asked to detect.

**Evidence requested by the brief, summarized:** users who complete actions through the *correct* UI (Beast/Guidance modes) do get evolving recommendations. Users who complete actions through Analysis mode, or who have an uncommon role at critical risk, get recycled content. This is a coverage gap, not an architectural failure — the mechanism for "evolve" already exists and works; it's just not applied everywhere.

## Section 9 — Intelligence Briefing Audit

**Score: 80/100** — the strongest section of the product.

The dashboard already follows almost exactly the Situation → Threats → Opportunities → Future → Protection → Actions structure the brief asks for, just under different labels:

`Situation Brief` → `Main Concerns` (ranked, scored) → `Your Escape Route` / `Career Escape Routes` → `Your Odds — Next 12 Months` (quantified with/without-action comparison, e.g. "22% vs 15%") → `Career Preparedness` + `5-Pillar Breakdown` → `Career Contingency Plan` + `Action Roadmap`.

This is genuinely a briefing, not a report — see the full sequence in [07](./ux-audit-screenshots/07-dashboard-situation-brief.png) through [12](./ux-audit-screenshots/12-dashboard-action-roadmap-debug-leak.png). The "How the AI reasoned" disclosure with a confidence percentage is exactly the kind of transparency move that separates an intelligence product from a scored quiz.

**What's holding it back from 90+:** this structure is invisible until after the wizard. None of Home/Pricing/Products previews this briefing format, so it can't do its job of setting expectations before the user invests 8 steps + a ~70s wait. A single "sample briefing" preview on the homepage (using static/cached data) would let the format sell itself before the CTA.

## Section 10 — Dark Mode Audit

**Score: 52/100**

The dark token system itself is excellent: 195 custom properties, `oklch()` color space, a real elevation/shadow/glow system (`index.css:4-292`). The problem is **light mode is an incomplete second coat of paint, not a parallel system**: of those 195 root variables, only ~70 are redefined under `html.light` (`index.css:8837-9307`). Confirmed missing from light mode: the entire elevation/shadow system (`--elevation-1..4`, `--shadow-elev-1..4`), the 8-color chart palette (`--chart-1..8`), risk-tier gradients (`--gradient-risk-*`), glow-card tokens, and the score-ring end color. Components keyed to these render their dark-tuned (often white-glow) values verbatim on a white background.

This is visible and reproducible: [03-wizard-light-mode-bug.png](./ux-audit-screenshots/03-wizard-light-mode-bug.png) shows the actual captured bug — toggling to light mode on `/terminal` turns the page background white, but the wizard card itself stays **pure black**, because its background is one of the 1,152 inline `color:`/`background:` literals (rgba/hex) that never reference a CSS variable at all (worst offenders: `StrategyTab.tsx` 49, `BehavioralIntelligencePanel.tsx` 43, `JobMarketLiquidityCard.tsx` 31 occurrences). 140 of these 1,152 are text colors at ≤0.30 opacity white — a contrast failure in dark mode already, and functionally invisible on a light card.

**Verdict:** dark mode, as the default and primary mode, is well executed. Light mode is not production-ready — it should either be finished (redefine all 195 tokens, migrate the worst-offender files to var() references) or removed from the toggle until it is, because a half-broken light mode is worse for trust than not offering one.

## Section 11 — Mobile Experience Audit

**Score: 64/100**

Navigation is well-architected for thumb use: the top nav is fully hidden below 768px (`index.css:2143-2145`) in favor of a fixed bottom tab bar and a "more" sheet, both within thumb reach (confirmed live: [04](./ux-audit-screenshots/04-home-mobile-dark.png), [05](./ux-audit-screenshots/05-mobile-more-sheet.png)). The actual completed dashboard adapts its tab bar to a fixed bottom strip too ([19-dashboard-mobile-top.png](./ux-audit-screenshots/19-dashboard-mobile-top.png)), and swipe-to-change-tab is implemented and wired (`useSwipeGesture.ts:24-58` → `LayoffAuditDashboardV3.tsx:60,554-563,752`).

Specific issues found:
- **Touch targets under 44px:** `.theme-toggle` (40×40px, `index.css:2110-2113`), the mobile-header settings avatar (34×34px, `App.tsx:291-292`), the mobile-header sign-in button (34px tall, `App.tsx:309`).
- **Architectural dead code:** `.nav-mobile-actions` (the element containing those three undersized targets) is styled to appear at ≤768px (`index.css:3106-3109`), but its parent `.nav-root` is forced `display:none !important` at the same breakpoint (`index.css:2143-2145`) — so this isn't "reachable but small," it's unreachable. Either the breakpoints disagree by accident or this is genuinely dead CSS.
- **Text truncation on mobile:** captured live in [20-dashboard-mobile-escape-route-truncation.png](./ux-audit-screenshots/20-dashboard-mobile-escape-route-truncation.png) — "How the AI re…" and "Advance to a Senior/Lead Role Before the Window C…" both clip mid-word with no "show more," losing the exact reasoning the desktop "How the AI reasoned" disclosure is built to surface.
- **Un-responsive grids:** `.survival-horizon-strip` (`index.css:7819-7822`) and the duplicated `.profile-reprompt-grid` rule (`index.css:6736-6741` and `7226-7232`) are both fixed multi-column with no mobile override, unlike `.audit-field-row`, which correctly collapses at ≤480px — the responsive pattern is known, just inconsistently applied.

## Section 12 — Emotional Experience Audit

**Score: 68/100**

**Emotional journey, first-time critical-risk user:**
1. **Curiosity → mild anxiety** (Home headline "Know You're At Risk Before The Layoff Happens" — direct, slightly alarmist, effective hook)
2. **Investment** (8-step wizard, well-paced, each step has a single clear question — [06](./ux-audit-screenshots/06-wizard-step3-radio-cards.png))
3. **Suspense** (the particle/network pipeline visualization during the ~60-75s analysis wait is genuinely well-crafted — see Section 13)
4. **Validation, not alarm** (the score reveal leads with a Situation Brief framed as "manageable today — build buffer before they compound," not a scare number — this is good emotional design; it informs without panicking)
5. **Direction** (Career Contingency Plan + Action Roadmap give the user something to *do*, which converts anxiety into agency — this is the single most important emotional move the product makes)
6. **Mild deflation on return** (Settings shows 0/14 achievements, all locked with "???" — fine on first visit, but if the mission-system gaps in Section 7 mean a returning user doesn't see progress, the "agency" from step 5 doesn't get reinforced)

**Net read:** a user finishes the *audit flow* feeling informed and motivated, not scared — that's a real achievement for a "you might lose your job" product. The risk is entirely in *return* visits, where the Section 7/8 personalization gaps mean the system may not visibly reward the actions the first session convinced the user to take.

## Section 13 — Premium Experience Audit

**Score: 71/100**

Framer Motion is not a thin showcase layer — it's used 881 times across 183 files, with the dashboard content layer (`src/components/AuditTabs/common/*`) alone accounting for 351 uses, more than the nav (5 uses in `App.tsx`). The hypothesis that polish is concentrated only in the nav pill is **not supported** by the evidence. What *is* true is that polish is qualitatively uneven: the nav pill and score hero get bespoke spring physics, parallax (`Tilt3D`), and animated SVG arcs, while most dashboard panels reuse the same generic `fadeInUp` — functional, slightly repetitive, not a "showcase" problem.

Strong, specific wins:
- The pipeline-progress visualization during audit analysis (a live particle/node network with percentage readout) is a genuinely premium loading state — most products would ship a spinner here.
- `BrandedLoader.tsx` + matched skeleton components (`SummaryTabSkeleton`, `CompanyTabSkeleton`, `CardSkeleton`) are wired into every major async boundary.
- `canvas-confetti` fires on score-improvement re-audits (`ScoreImprovementCelebration.tsx:28-50`) — a real micro-delight moment.

Gaps:
- Confetti/celebration is wired to exactly **one** moment (score improvement). First-audit completion and action-plan milestones get a plain toast, not the celebratory treatment — an easy, low-risk expansion of an effect that already exists and works.
- `react-loading-skeleton` is an installed dependency with **zero imports anywhere in `src`** — the team built a better custom skeleton system and never removed the now-redundant package.
- 39 hover effects across 18 files use raw `onMouseEnter`/`onMouseLeave` JS state instead of CSS `:hover`, adding unnecessary re-render/jank risk on lower-end devices for what's almost always just a tooltip toggle.
- The "System Override Applied" debug-string leak (Section 7) directly undermines "polished" in the one place users are reading most carefully — the action plan.

## Section 14 — Visual Intelligence Audit

**Score: 68/100** — substantially better than the brief's hypothesis predicted.

The named candidate charts (`RiskTrendChart`, `SkillRadarChart`, `SkillDependencyGraph`, `CareerPathMap`, `SocialProofStrip`) are **not orphaned** — all five are imported and rendered in live code paths (cited in Section 3). The real finding is two-fold:
1. The richest chart (`RiskTrendChart`) is gated behind `history.length >= 1`, so the highest-value first-time audience never sees it (Section 3).
2. The app runs **two parallel charting approaches** simultaneously — `recharts` (used in `ScoreDriftTracker.tsx`, `SkillRiskCalculator.tsx`, `HumanIrreplacibilityIndex.tsx`, `DisplacementForecast.tsx`) and hand-rolled inline SVG (the v3-dashboard chart set). Neither is wrong, but it's an architectural seam that will show up as subtle visual-language drift between, e.g., the Risk Calculator's charts and the Audit dashboard's charts.

Remaining text-that-should-be-visual: the 5-dimension risk-driver distribution (`SummaryTab.tsx:655-672`, only top-3 shown as text cards) and equity/vesting figures (`OverviewTab.tsx:1837,1843`, bare numbers with no timeline).

### The one-paragraph verdict

The **post-audit dashboard** — the thing a user sees after 8 steps and a ~70-second wait — is the best work in this codebase and already clears the "personal career intelligence agency" bar the brief sets: Situation Brief, AI Read with a reasoning trace, named strategic paths with feasibility math, quantified odds. That experience is undermined by three fixable things, not a design-philosophy problem: (1) a visible debug-string leak and a disconnected duplicate completion-tracking UI in the Action Plan, which directly contradicts "the system evolves"; (2) a light mode that is an unfinished second coat of paint over 30% of the design tokens, producing a literal black-card-on-white-page bug; and (3) marketing/utility pages (Home, Risk Calculator, Products, Team) that don't carry any of the dashboard's voice or density discipline, so the product undersells itself before the CTA. None of these require new features — they require finishing work that's already 80% done.

### Top 5 fixes by impact-to-effort ratio
1. Rewire `ActionPlanTab.tsx`'s checkbox (`:1999,2057`) to call `markActionComplete` — fixes the most damaging "recycled intelligence" bug for near-zero risk.
2. Remove or rewrite the "System Override Applied" debug string in the Action Roadmap.
3. Replace wizard + achievement emoji with the already-dominant lucide set (mapping fully specified in Section 4) — single afternoon, highest visible-polish-per-hour fix available.
4. Wire the 6 unused `AdditionalIllustrations.tsx` components into the empty-state/text-heavy spots enumerated in Section 5 — assets already exist, zero design cost.
5. Either complete the `.light` token set in `index.css` or hide the theme toggle until it is — the current state is a visible, reproducible bug ([03-wizard-light-mode-bug.png](./ux-audit-screenshots/03-wizard-light-mode-bug.png)).
